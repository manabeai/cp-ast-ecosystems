# サブプロジェクトA: AtCoder調査 + E2Eテスト拡充 設計書

## 概要

既存22個のE2Eテスト（5カテゴリ）が実際のAtCoder ABCの入力パターンを十分にカバーしているか検証し、不足があれば新しいE2Eテストを追加する。

**スコープ**: A→B→Cの3サブプロジェクト構成の最初のフェーズ。ここでE2Eテスト仕様を確定し、後続のB（バックエンドAPI完成）とC（フロントエンド実装）の要件を固める。

## 既存E2Eテストカバレッジ

| カテゴリ | specファイル | テスト数 | 対応する入力パターン |
|---|---|---|---|
| basic-array | basic-array.spec.ts | 5 | N / A_1...A_N |
| grid | grid.spec.ts | 4 | H W / S_1...S_H |
| tree | tree.spec.ts | 4 | N / u_1 v_1...u_{N-1} v_{N-1} |
| query | query.spec.ts | 4 | N Q / variant分岐 |
| multi-testcase | multi-testcase.spec.ts | 5 | T / (N / A_1...A_N) × T |

## 調査方法論

### 対象

- AtCoder ABC の直近の問題（ABC350〜ABC410 あたりから約50問をサンプリング）
- A〜G問題を含み、難易度による入力形式の多様性を確保

### 手順

1. **問題収集**: サブエージェントが AtCoder 問題ページにアクセスし、各問題の入力形式セクション（「制約」「入力」欄）を抽出
2. **パターン分類**: 各問題の入力形式を既存カテゴリに当てはめる
   - `scalar` / `array` / `grid` / `tree(edge-list)` / `query(choice)` / `multi-testcase`
   - 組み合わせパターン（例: multi-testcase + tree）も記録
3. **新パターン候補の収集**: 既存カテゴリに分類不能な問題を収集
4. **汎用性評価**:
   - 3問以上で類似パターン → 汎用パターンとして採用し、E2Eテスト作成
   - 1〜2問のみ → 問題固有として却下
   - ただし、汎用パターンでも「このパターン専用のAPI」が必要になるなら却下

### 期待される新パターン候補（仮説）

調査前の予測として、以下のパターンが見つかる可能性がある：

- **重み付き辺リスト**: `u_i v_i w_i` — edge-list の変種（NodeKind は同一、属性追加）
- **2次元座標**: `x_i y_i` — 同一行 scalar × 2 の配列
- **文字列配列**: `S_1...S_N` — array with string type
- **複数配列**: `A_1...A_N` と `B_1...B_N` — 複数の独立した配列

## E2Eテスト追加方針

### 制約

- 既存22テスト（5 specファイル）は **一切変更しない**
- `editor-page.ts` の既存メソッドは **変更しない**（新メソッド追加のみ）
- 追加テストは新しい spec ファイルとして作成
- 新パターンごとに最低2テスト: 構造構築テスト + 完成状態検証テスト

### 成果物

- 新specファイル群（0〜数個。調査結果次第）
- `editor-page.ts` への新メソッド追加（必要な場合のみ）
- `README.md` の testid 契約追加（既存削除なし）
- パターン分類結果レポート（セッション内で保持、コミットしない）

## サブエージェント構成

### フェーズ1: 問題収集（並列可能）

- **問題収集エージェント**: AtCoder ABC問題の入力形式をWeb取得
  - 入力: 対象コンテスト範囲（ABC350〜410）
  - 出力: 問題ごとの入力形式テキスト + 制約テキスト

### フェーズ2: ギャップ判定（収集結果依存）

- **ギャップ判定エージェント**: 分類結果と既存E2Eの差分分析
  - 入力: 分類表 + 既存22テストの仕様
  - 出力: 新テスト仕様リスト（新パターン名、対応問題、テスト要件）

### フェーズ3: テスト実装（ギャップ判定結果依存）

- **テスト実装エージェント**: 新specファイル・POMメソッド追加
  - 入力: 新テスト仕様
  - 出力: specファイル群 + POM更新

## 成功基準

1. 直近50問の入力形式が全て分類済み
2. 汎用的な新パターン（3問以上出現）にはE2Eテストが追加されている
3. 既存22テストが変更されていない
4. Playwright `--list` で全テスト（既存+新規）が検出される
5. テスト仕様が後続サブプロジェクトB・Cの要件として使える状態
