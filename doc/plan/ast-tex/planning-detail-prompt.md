# cp-ast-ecosystems 詳細設計の進め方指示書

## 目的

本指示書は、`cp-ast-core` を土台として、

- 制約の表記
- 入力の表記
- 将来的なランダムテスト生成
- wasm / js / ts 連携

へ進むための**詳細設計の進め方**を定義するものである。

この文書の主眼は、単に理想設計を書くことではない。  
むしろ重要なのは、**現状の既存実装**と**これまでに書かれた設計指示書**のあいだにある乖離を発見し、

- 既存実装に合わせて設計を修正すべきか
- core に新しい概念や補助型を追加すべきか
- 単に指示書の想定が古いだけか

を切り分けながら、壊さずに前進することである。

---

## 最重要方針

詳細設計は、必ず次の順番で進めること。

1. **既存実装を読む**
2. **既存実装から現在の事実を抽出する**
3. **既存実装と設計文書の差分を明示する**
4. **差分ごとに、どちらを直すべきか判断する**
5. **その判断理由を明文化する**
6. **そのうえで初めて詳細設計を書く**

つまり、最初に理想設計を押しつけて既存コードを破壊しないこと。

---

## 前提認識

### 中核

- `cp-ast-core` はすでに存在する
- それは競プロ入力仕様の意味構造を保持する中核である
- AST と制約の整合性が保証されれば、派生物はバックエンドで演繹できる

### バックエンドで演繹したいもの

- 制約の表記
- 入力の表記
- ランダムテスト生成機

### フロントエンド方針

- 自由テキスト入力ではなく GUI ベースの構造化エディタ
- GUI が不正状態を構築できないようにする
- そのために `projection` と `operation` がある

### projection の役割

- 現在の AST から遷移可能な合法状態の集合を返す
- フロントはその結果を表示するだけに寄せる
- フロントが複雑な整合性ロジックを持たなくてよいようにする

### operation の役割

- AST に特定の操作を適用し、更新後 AST を返す
- 状態遷移をバックエンド側で支配する

### 現在の主ターゲット

現時点では、まず以下を作ることが主目的である。

- **制約の表記**
- **入力の表記**

自然文の Human モード全文生成を先にやることは主目的ではない。  
必要なのは、競プロらしい安定した pretty printer / TeX renderer である。

## 詳細設計の進め方

## Step 1. 実装事実の棚卸し

まず、既存リポジトリから**現在すでに実装されている事実**を抽出する。

少なくとも次を確認すること。

### core
- structure 周りの型
- constraint 周りの型
- reference / expression
- projection api
- operation / action
- render にすでに何があるか
- tests が何を前提にしているか

### 実装事実として明示すべきもの
- 現在存在する enum / struct / trait
- すでに確定している責務境界
- 命名
- すでにテストで固定化されている挙動
- コメントや docstring による意図
- まだ未実装だが名前だけある概念

この段階では提案をしない。  
まずは「いま何があるか」を書き出す。

---

## Step 2. 既存実装から暗黙の設計を抽出する

実装には、設計文書に明示されていなくても暗黙のルールが埋まっている。  
それを取り出す。

例:

- `Constraint::RenderHint` が存在するなら、render 情報の一部は constraint 側責務という設計である
- `StructureAst` が insertion order を保持しているなら、canonical order を重視している
- `ProjectionAPI` に `hole_candidates` があるなら、hole は第一級概念である
- `Action` が low-level なら、UI / Agent 操作は low-level command ベースで考えている

ここでは、既存コードから**「実装済み設計思想」**を抜き出す。

---

## Step 3. 既存実装と既存指示書の乖離を洗い出す

次に、これまで書かれた設計文書や指示書と、既存実装を比較する。

乖離は最低でも次の4種類に分類する。

### A. 指示書の理想が強すぎるだけ
例:
- 指示書では generator / tex / wasm / json / ts を綺麗に分けているが、既存実装ではまだそこまで分割されていない
- しかし core の責務上、今すぐ分けなくてもよい

### B. 既存実装が設計意図からずれている
例:
- 本来 constraint 側にあるべきものが structure 側に入り込んでいる
- 本来 projection で返すべき情報が operation 側前提になっている

### C. core に概念が足りない
例:
- TeX 化に必要な最小限の index variable の表現が core にない
- 入力表記の安定生成に必要な slot 情報が足りない
- 配列要素の代表表現を安定に取るための補助情報が不足している

### D. 指示書のほうが古い / 不正確
例:
- 既存 core がすでに新しい責務分離を採用している
- 指示書の想定が昔の構成に引っ張られている

---

## Step 4. 乖離に対してどう対処するか決める

差分を見つけたら、必ず以下の三択のどれで処理するか決める。

### 1. 既存実装に合わせて設計文書を直す
使う条件:
- 既存実装の方が筋が良い
- テストや責務がすでに安定している
- 指示書だけが古い

### 2. core に最小限の追加を行う
使う条件:
- 今後の renderer / generator / wasm 境界に必要な概念が本当に不足している
- 追加によって core の意味モデルが自然に強化される
- 無理に外側で吸収すると不自然になる

### 3. 外側レイヤで吸収する
使う条件:
- core の意味を増やすほどではない
- purely presentational な都合である
- generator / tex / json DTO の層で十分処理できる

重要なのは、**なんでも core に入れない**こと。  
逆に、**本当に core にあるべきものを外側の泥臭い処理で誤魔化さない**こと。

---

## Step 5. 詳細設計を書く順番

詳細設計は次の順で進める。

### 5-1. 制約表記
最初にやるべき。  
理由:
- `Constraint` の情報量が比較的閉じている
- 出力形式が itemize 的で安定しやすい
- 競プロ問題での可視価値が高い

### 5-2. 入力表記
次にやる。  
理由:
- `StructureAst` の解釈が必要
- repeat / tuple / array / choice をどう TeX 化するかの規則が要る
- projection とも思想が近い

### 5-3. generator / tex の境界整理
制約表記と入力表記を書いてから整理する。  
最初から理想分割しすぎない。

### 5-4. json / wasm / ts の境界
最後に外部公開境界を詰める。  
内部責務が曖昧なまま wasm api を固定しない。

---

## 詳細設計で必ず扱う観点

各詳細設計タスクで、必ず次の観点を含めること。

### 1. 既存実装との整合
- いまの core でそのままできるか
- 追加が必要なら何を足すか
- 追加しない場合は外側でどう吸収するか

### 2. 差分安定性
- 同じ AST なら同じ出力になるか
- golden test が書けるか

### 3. 失敗時の挙動
- hole を含むときどうするか
- unsupported な制約が来たときどうするか
- warning を返すか、error にするか

### 4. 表記ルールの局所化
- 変数の添字表現
- index range 表現
- `\cdots`, `\vdots`
- `\times`, `\le`

が ad-hoc に散らばらないようにする

### 5. 将来のランダムテスト生成との関係
- 今の設計が generator を壊さないか
- presentational な都合を core に入れすぎていないか

---

## 批判的 Subagent を必ず入れる

詳細設計では、設計を書く agent とは別に、**批判専門の Subagent** を必ず入れる。

この Subagent の役割は「いい感じにまとめる」ことではない。  
むしろ次を徹底すること。

- 設計が既存実装を無視していないか
- 既存テストと衝突しないか
- core に不要な責務を押し込んでいないか
- 逆に本来 core にあるべきものを外側で誤魔化していないか
- API が早すぎる固定化になっていないか
- wasm / dto 境界が premature でないか
- TeX 都合が core を汚していないか

### 批判的 Subagent の出力形式

必ず次の形式で出力させる。

1. **実装無視の懸念**
2. **責務分離の破綻点**
3. **core に足すべき / 足すべきでない点**
4. **今 fix すべき点**
5. **後回しでよい点**
6. **最も危険な設計判断トップ3**

---

## 推奨 Subagent 構成

最小でも次の構成を推奨する。

### Subagent 1: Current-Implementation Reader
役割:
- リポジトリを読んで現状実装を整理する
- 実装済み責務と暗黙設計を抽出する

### Subagent 2: Gap Analyzer
役割:
- 実装と指示書の乖離を一覧化する
- 差分を A/B/C/D 分類する

### Subagent 3: Constraint Renderer Designer
役割:
- 制約表記の詳細設計を書く
- 既存 constraint 型との整合を確認する

### Subagent 4: Input Renderer Designer
役割:
- 入力表記の詳細設計を書く
- structure の repeat / tuple / array をどう解釈するか明文化する

### Subagent 5: Boundary Designer
役割:
- json / wasm / ts の境界設計を書く
- ただし内部責務を壊さない範囲に留める

### Subagent 6: Critical Reviewer
役割:
- 全体を批判的に見る
- 「理想だが危険」「今やる必要がない」「既存実装と衝突する」を強めに指摘する

必要なら最後に

### Subagent 7: Arbiter / Synthesizer
役割:
- 各 Subagent の結論を統合する
- 採用案 / 見送り案 / open question を整理する

を置く。

---

## 各 Subagent に共通で守らせること

すべての Subagent は次を守ること。

### 1. まず既存実装ベースで話す
理想論から始めない。  
いまの repo に存在する型と責務を前提にする。

### 2. 「こうあるべき」だけで終わらせない
必ず次のどれかを明示する。

- 既存実装に合わせる
- core に追加する
- 外側レイヤで吸収する

### 3. 変更コストを評価する
- テスト破壊の可能性
- 命名変更のコスト
- wasm / dto 公開境界への影響

### 4. 早すぎる一般化を避ける
- render-ir の独立
- 過剰な visitor 抽象
- 早すぎる plugin 化
- 不要な汎用 API 化

を避ける。

### 5. 不確実な点を曖昧にしない
わからないことは「未確定」として列挙する。  
勝手に補完しない。

---

## 詳細設計タスクの推奨順

### Task 1. 現状実装監査
成果物:
- 現在ある型・責務・暗黙ルール
- 指示書との差分表

### Task 2. 制約表記詳細設計
成果物:
- 制約ごとの TeX 表記規則
- warning / unsupported policy
- golden test 案

### Task 3. 入力表記詳細設計
成果物:
- scalar / tuple / array / repeat / choice の TeX 表記規則
- hole の表示規則
- golden test 案

### Task 4. core 拡張要否レビュー
成果物:
- core に足すべき最小追加案
- 足さずに済むもの一覧
- 理由付き判断

### Task 5. 境界 api 詳細設計
成果物:
- dto
- envelope
- wasm の json string api
- ts sdk の最小 api

### Task 6. 批判レビュー
成果物:
- 危険な設計判断
- 今 fix すべき点
- 後回し項目
- open questions

---

## 判断のための基準

詳細設計の各段階で、次の基準を使って判断すること。

### core に入れる基準
- 意味モデルとして自然
- renderer だけでなく generator や projection にも利く
- ad-hoc な外側処理では不自然になる

### 外側で吸収する基準
- purely presentational
- TeX / dto / wasm だけの都合
- 競プロ表記の流儀差にすぎない

### 指示書を直す基準
- 既存実装の方がすでに筋がいい
- 指示書が理想化しすぎている
- 今の設計判断を無理に崩す必要がない

---

## 最終成果物のあるべき形

最終的な詳細設計成果物は、単なる夢の設計ではなく、必ず次を含むこと。

1. **現状実装の要約**
2. **現状実装と指示書の乖離一覧**
3. **各乖離への対処方針**
4. **採用案**
5. **見送り案**
6. **core に足す候補**
7. **外側レイヤで吸収する候補**
8. **テスト方針**
9. **open questions**
10. **批判的レビュー結果**
