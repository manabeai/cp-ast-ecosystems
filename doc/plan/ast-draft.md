# 目的

本ライブラリの目的は、ユーザやAIが構造を編集・構築する過程において、不正な状態を可能な限り構築不能にしつつ、未完成状態も安全に表現できる抽象構文基盤を提供することである。

従来の多くのASTは、パース後の解析対象として設計される。
本ライブラリではそれに対し、ASTを

解析結果
ではなく
構築・編集の中心モデル

として扱う。

そのため本ライブラリは単なる木構造ではなく、以下を備える。

構造の表現
制約の表現
未完成状態の表現
編集可能性の表現
文脈に応じた候補列挙の基盤

## 中核理念
### 構造は文字列より先にある

本ライブラリでは、テキスト表現やGUI表現は本質ではない。
本質は 構造そのもの である。

テキストは構造の一つの入出力形式
View は構造の一つの投影
AIによる提案も構造への操作列として扱う

したがって、正本は常にASTである。

### 不正状態は事後検出より事前排除を優先する

一般的な編集系は「自由に書かせて後でエラーにする」。
本ライブラリはできる限り

そもそも作れない
少なくとも安全な未完成状態としてしか存在できない

ことを目指す。

ただし、すべてを静的型で表現するのではなく、

型で排除できるものは型で排除
文脈依存なものは制約として表現

する。

### 未完成状態は第一級の存在である

編集・生成・補完の過程では未完成状態が本質的に存在する。
このため hole は例外ではなく、構造モデルの第一級要素として扱う。

ただし hole は単なる空欄ではなく、

その位置に要求される型
その位置に要求される構造上の役割
その位置で許される候補や操作

を導出するための制約付き未完成ノードである。

### 構造と制約は分離しつつ一貫して扱う

本ライブラリでは

StructureAST: 何がどのように組み合わさっているか
ConstraintsAST: その構造にどのような制約が課されているか

を分離する。

これにより、

構造自体は比較的安定に保つ
制約の表現を拡張可能にする
候補列挙や操作可能性の導出を柔軟にする

ことを狙う。

### 主要構成

本ライブラリのコアは大きく次の二層からなる。

StructureAST
  構造の骨格を表す

ConstraintsAST
  構造に課される制約を表す

その上に将来的に

Operation layer
Projection layer
Import / Export layer

が載る。

ただし本資料ではコアである StructureAST / ConstraintsAST に絞る。

#### StructureAST の基本理念
### 役割

StructureAST は「何がどこにあるか」を表す。
これは構文木に近いが、通常のASTよりも編集可能性を意識した骨格である。

StructureAST は以下を担う。

ノード種別
親子関係
スロット構造
部分木の接続
hole の配置位置

StructureAST 自体は、できるだけ

シンプル
永続的
比較可能
シリアライズ可能

であるべき。

### StructureAST は「木」ではなく「構造編集可能な状態空間」

通常のASTは解析結果としての木で十分だが、本ライブラリでは編集が前提である。
そのため StructureAST は単なる木というより、

現在の構造状態
そこから可能な次状態への基盤

と見る。

ただし「遷移そのもの」は StructureAST ではなく後段の操作意味論が担う。
StructureAST 自体はあくまで状態表現である。

### スロット中心設計

親ノードは単に子ノードを雑に配列で持つのではなく、意味を持ったスロットを持つ。

例:

If ノード
cond
then_branch
else_branch
BinaryOp ノード
lhs
rhs

この設計により、

hole がどの役割の欠損か明確になる
ConstraintsAST 側で slot ごとの制約を自然に記述できる
View への投影が容易になる
### hole は StructureAST 上の正規ノード

hole は null ではなく、正規のノードまたはスロット値として扱う。
ただし重要なのは、hole の実体は「何もないこと」ではなく、

ここが未完成である
ここには何らかの制約がある

という事実である。

StructureAST 単独では hole は「未充足な位置」であり、その意味付けは ConstraintsAST で補う。

### NodeId と参照安定性

編集や差分比較を行うため、各ノードには安定した NodeId を付与する。
これにより、

部分更新
操作履歴
View との対応
AI提案対象の特定

が可能になる。

NodeId はあくまで識別子であり、意味は持たない。

### 構造と表示を分離する

StructureAST は View の都合を直接持たない。

持つべきでないもの:

色
レイアウト
DOM構造
UIコンポーネント種別

持ってよいもの:

ラベル生成に必要な最小限の情報
スロット名
ノード種別
役割名

#### ConstraintsAST の基本理念
### 役割

ConstraintsAST は「この位置に何が許されるか」を表す。
StructureAST が骨格であるのに対し、ConstraintsAST は妥当性条件の宣言的表現である。

ConstraintsAST は次を担う。

型制約
スロット制約
演算子適用制約
文脈依存の候補制約
将来的にはスコープや能力制約
### 制約は「エラー判定」ではなく「可能性の定義」

ConstraintsAST の目的は、違反を見つけて怒ることだけではない。
より重要なのは、

何が置けるか
どんな候補があり得るか
どんな操作が可能か

を導出することにある。

つまり ConstraintsAST は禁止条件の集合ではなく、構築可能性の仕様である。

### 型制約と文脈制約を統一的に扱う

制約には大きく二種類ある。

型レベルで比較的静的なもの
if 条件は Bool
+ の左右は Int
Hole<Int> に Bool は入らない
値レベルで文脈依存なもの
現在のスコープで見える変数のみ
この文脈では副作用禁止
比較可能型のみ許可
この DSL フェーズではこのノードは禁止

本ライブラリでは前者を型システムで、後者を ConstraintsAST で扱う。
ConstraintsAST は特に後者の表現に強みを持つ。

### 制約はスロットや hole に紐づく

ConstraintsAST はノード全体にかかることもあるが、基本的には

hole
slot
ノード位置
親子関係

に紐づく。

これにより

hole に入れられる候補一覧
slot に要求される期待型
その位置で可能な編集操作

が自然に導出できる。

### 制約は合成可能であるべき

一つの位置には複数の制約が同時にかかる。

例:

型は Int
比較可能である
副作用を持たない
現在のスコープから参照可能

したがって ConstraintsAST は単一の型情報ではなく、制約の合成を表現できる必要がある。

制約は概念的には

候補集合 = 型制約 ∩ 構造制約 ∩ 文脈制約 ∩ 能力制約

として働く。

### hole の期待型は制約の一部であって全体ではない

hole には期待型があるが、それだけで十分ではない。
たとえば Int hole であっても、

比較用の Int
加算用の Int
ループ境界用の Int

のように意味役割が異なる場合がある。

よって ConstraintsAST は期待型だけでなく、

slot role
required capability
contextual predicate

を表現できるべきである。

#### StructureAST と ConstraintsAST の関係
### StructureAST は形、ConstraintsAST は許容範囲

両者の関係は次のように整理できる。

StructureAST:
現在の形
ConstraintsAST:
その形の各位置に対して何が許されるか

この分離により、

StructureAST 自体はシンプルに保ちやすい
制約拡張の影響を局所化できる
候補導出ロジックを整理しやすい
### StructureAST だけでは編集不能

StructureAST 単体では「未完成の位置がある」ことしかわからない。
そこに何を入れるべきかは ConstraintsAST がないと判断できない。

このため GUI や AI が hole を埋めるには必ず ConstraintsAST 由来の情報が必要である。

### ConstraintsAST だけでは実体がない

逆に ConstraintsAST 単体では、どこに何が既に存在しているかがわからない。
したがって、両者は常に

StructureAST が実体を持ち
ConstraintsAST が許容範囲を与える

という関係にある。

#### 型システムとの役割分担
### 型で守るもの

ライブラリ全体では phantom type 等を用いて、以下のような大きな不正を型で防ぐ。

Expr<Int> と Expr<Bool> の混同
If 条件に Bool 以外が入る
Add の左右に Int 以外が入る
Hole<T> に異なる型の式を埋める

これにより構造の骨格に関する誤りはコンパイル時に落とす。

### ConstraintsAST で守るもの

一方、以下は型だけでは表現しきれないので ConstraintsAST 側に寄せる。

スコープに応じた候補
使用可能な識別子一覧
比較可能性などの能力
文脈依存の禁止事項
編集途中の局所ルール

この役割分担により、型システムを過剰に複雑化せずに済む。

#### 設計方針
### Core first

コアはあくまで StructureAST / ConstraintsAST である。
View や parser は境界機能として後付け可能にする。

優先順位は次の通り。

StructureAST
ConstraintsAST
操作意味論
Projection
Import/Export
### Text-first ではなく Structure-first

文字列から parse できることは有用だが、本設計の中心ではない。
文字列表現は StructureAST の一つの import/export 形式に過ぎない。

### GUI-first ではなく Editable-core first

本設計はGUI利用を強く意識しているが、GUI専用にはしない。
同じコアが将来的に

WebUI
CLI
AI Agent
テスト生成器

から利用できるべきである。

### View は projection であり正本ではない

View は StructureAST と ConstraintsAST から導出される像である。
View 側に制約判断や妥当性判定を持ち込まない。

### hole は GUI のための実用概念であり、コアでも第一級に扱う

特にGUI編集では hole がないと

どこが未完成か
何を置けるか
次に何ができるか

をUIが知れない。
このため hole は View 都合ではなく、コア概念として扱う。

ただし文字列入力では hole を必須にしない運用も可能とする。

### 宣言的制約を優先する

制約は可能な限り宣言的に記述されるべきである。
候補列挙や操作可能性は、その宣言から導出できる形が望ましい。

### 差分更新と永続性を意識する

編集過程を扱う以上、部分木差し替えや履歴管理を見据える。
そのためノード識別と部分更新に向いた構造を採用する。

#### 想定される責務分離
StructureAST の責務
ノード種別の表現
親子・スロット関係
hole の位置表現
ノード識別
部分木の保持
ConstraintsAST の責務
期待型
slot role
能力制約
文脈依存候補の制約
hole / slot ごとの許容条件
後段の操作層の責務
可能操作の列挙
操作適用
制約検査
エラーレポート

#### 最小実装で最初に必要なもの

最小の設計対象としては次を推奨する。

StructureAST
NodeId
ノード種別 enum
slot 概念
hole ノード
typed expression skeleton
ConstraintsAST
expected type
slot role
allowed node kinds
optional contextual predicates

この時点ではまだ

parser
rich view
AI補完
複雑なスコープ解析

は後回しでよい。

#### View への言及

View は本ライブラリの主役ではない。
ただし応用例として、StructureAST と ConstraintsAST から以下を導出できる。

現在の構造の表示単位
hole の位置
各 hole / slot に対して許される候補
実行可能な操作一覧

この意味で View は

構造と制約の投影結果

であり、独自の意味論を持たない薄い層として設計する。

#### まとめ

本ライブラリの核は、通常の「解析のためのAST」ではなく、

編集・構築・補完のための抽象構文基盤

を作ることにある。

そのために、

StructureAST が構造の骨格を表し
ConstraintsAST がその位置ごとの許容範囲を表し
hole が未完成状態を第一級に表現し
型システムと制約システムを分担させる

という方針を採る。

一言で言えば、このライブラリは

「妥当な構造だけを安全に構築・編集するためのコア」

である。

ConstraintsAST / StructureAST 設計資料 追補
競プロDSLとしての位置づけと、制約・入力方式・出力可能性・サンプル生成保証

#### ドメイン前提
### 本ライブラリは競技プログラミング問題記述DSLを対象とする

本ライブラリが最初に対象とするドメインは、競技プログラミング問題の問題記述である。
ここでいう問題記述とは、主に次の要素を指す。

入力方式
出力方式
制約
サンプルケース生成の前提となる構造
問題文中で暗黙に要求される構造的関係

本ライブラリは、競技プログラミング問題を単なる自然言語の説明文としてではなく、構造化された仕様オブジェクトとして扱う。

### 競プロドメインを採用する理由

競技プログラミング問題は、研究対象として次の性質を持つ。

# 入出力仕様が明確
入力形式
出力形式
制約
が比較的定型的である。

## 正しさ判定がしやすい
生成されたサンプルケースが制約を満たすか
ASTから再構成された表記が元仕様と一致するか
を検証しやすい。

### 構造化DSLへの落とし込みに適している

競プロ問題は、自然言語で書かれていても実際には

配列
長さ
整数範囲
依存関係
複数テストケース
といった構造的要素の組み合わせで成り立つ。

#### 一般化可能な“仕様記述問題”の縮図である

競プロ問題はニッチな応用先ではなく、

制約付き入力仕様
宣言的フォーマット記述
妥当な例生成
という一般的な仕様モデリング課題の、評価しやすい具体例として位置づける。

#### このDSLの主対象
### 問題文全体ではなく、「入力・制約・生成可能性」を中心に扱う

本DSLの中心は、競プロ問題文全文の意味理解ではない。
主に対象とするのは以下である。

InputFormat
OutputFormat
Constraints
それらから導出される SampleCaseGenerator

つまり本DSLは「問題文生成DSL」ではなく、まずは

競技プログラミング問題の形式仕様DSL

として位置づける。

自然言語による説明文は将来的に補助的に扱う可能性はあるが、初期段階の正本ではない。

#### 競プロサイト上の「制約」「入力方式」とAPIを近づける方針
### ユーザが認知している概念単位とAPI単位を近づける

競技プログラミングの問題作成者・読者が普段目にする概念は、主に次のようなものである。

N
A_1, A_2, ..., A_N
1 ≤ N ≤ 2 × 10^5
0 ≤ A_i ≤ 10^9
入力は以下の形式で与えられる
T 個のテストケース
各ケースごとに…

本ライブラリでは、内部ASTがどれだけ抽象的であっても、外部APIはできるだけこれらの概念と近い形を保つ。

つまり API は、コンパイラ内部用の低レベル木操作だけではなく、競プロの制約記述で人間が使う単位に対応したドメイン近傍APIを持つべきである。

### APIは競プロ記法のラッパーではなく、競プロ概念の構造化表現である

ここで重要なのは、APIが競プロサイトの文字列表現の単なるラッパーではないことである。

目指すもの
N is an integer
A is an integer array of length N
A[i] ranges from 0 to 10^9
there are T test cases

のような概念を、構造として表せること。

目指さないもの
最初から "1 <= N <= 10^5" という文字列をそのまま持つこと
"A_1 A_2 ... A_N" という文字列片をそのままASTの本体とすること

つまり、人間の見慣れた競プロ表記に近いAPIを持ちつつ、その実体は構造化ASTであることが方針である。

### 例: ドメイン近傍APIのイメージ

たとえば以下のような操作概念を提供する。

整数変数を定義する
配列変数を定義する
その配列長を他の変数に結びつける
値域制約を与える
入力順序を指定する
複数テストケース構造を定義する

これらは最終的には StructureAST / ConstraintsAST に落ちるが、利用者には競プロ問題の記法に近い感覚で扱えるようにする。

#### ASTから「制約」「入力方式」を一意に構築可能であること
### 本ライブラリは AST を正本とし、競プロサイト風表現は導出物とする

本ライブラリでは、

競プロサイトに表示される「制約」
「入力方式」
必要に応じて「出力方式」

は、ASTから決定的に再構成可能であることを目指す。

つまり、

StructureAST + ConstraintsAST
  ↓
canonical rendering
  ↓
競プロサイト風の制約表記 / 入力方式表記

という流れを持つ。

### 「一意に構築可能」の意味

ここでいう「一意」とは、意味的に同じASTに対して、定められたレンダリング規則に従えば常に同じ表記が得られることを意味する。

これは以下を含む。

制約の順序規則が定義されている
変数表示名が正規化されている
入力方式の展開規則が定義されている
配列や繰り返しの表示が決定的である

ただしこれは「自然言語の完全な自由記述」まで一意に復元することを意味しない。
対象はあくまで、競プロ問題でよくある

制約欄
入力形式欄
出力形式欄

のような構造的で定型的な表示部分である。

### canonical rendering の重要性

この一意性は以下の理由で重要である。

# ASTが正本であることを保証するため

ASTが正本である以上、外部表記はASTから再現可能である必要がある。

## GUIやAI編集結果の検証がしやすくなるため

編集後にレンダリング結果を確認しやすい。

### 差分比較しやすくなるため

同じ意味なのに表記ゆれで差分が壊れることを減らせる。

#### 研究上の評価対象になるため
ASTから競プロサイト風表記が再生成できるか
その再生成が安定しているか
を評価対象にできる。

#### 競プロASTからランダムサンプルケースを生成可能であること
### ASTは単なる表示用構造ではなく、生成可能性の基盤である

本ライブラリにおける AST は、問題文や制約欄を再表示するだけでなく、
その仕様に従った具体的入力例を生成する基盤でもある。

つまり AST は少なくとも以下の二つの方向に使える。

AST -> human-readable specification
AST -> concrete sample cases
### サンプルケース生成は ConstraintsAST の自然な応用である

ConstraintsAST は、禁止判定だけでなく「何が許されるか」を表現する。
この思想により、サンプル生成は副次的機能ではなく、制約記述から自然に導出される。

例:

1 <= N <= 10
A is array of length N
0 <= A[i] <= 100

という制約があれば、

N を範囲から選び
その長さに応じて配列を生成し
各要素を範囲内で選ぶ

ことで妥当なサンプルケースを構築できる。

### 「生成可能保証」の意味

ここでいう「ランダム sample case が生成可能保証される」とは、厳密には次の意味で定義するのがよい。

条件付き保証

AST に表現された制約系が充足可能であり、かつ生成器が対応する制約種別を十分実装しているならば、
その AST から制約を満たすランダムサンプルケースを生成できる。

つまり保証は無条件ではなく、

制約が矛盾していないこと
制約種別が生成器の対応範囲内にあること

を前提とする。

この点は研究上も正直に明記するべきである。

### 生成器の責務

サンプル生成器は少なくとも次を担う。

制約の充足可能性の前提チェック
入力構造に沿った値生成
変数依存の解決
例: 長さ N が決まってから A を生成する
値域制約の尊重
複数テストケース構造の展開
canonical input serialization
### 生成可能性を意識したAST設計が必要

単に制約文を木にしただけではサンプル生成は難しい。
そのため ConstraintsAST は、生成の観点から少なくとも次を表現しやすい必要がある。

依存関係
例: 配列長が N に依存
局所制約と全体制約の区別
生成順序の導出可能性
スカラーと列・行列・グラフなどの構造的違い

つまり本DSLは、読める制約表現であるだけでなく、生成可能な制約表現である必要がある。

#### 競プロDSLとしてのConstraintsAST / StructureAST の追加方針
### StructureAST は入力構造を表現する

競プロドメインでは StructureAST は主に以下を担う。

スカラー入力
配列入力
2次元配列
繰り返し入力
テストケース単位
変数定義順

つまり StructureAST は「入力形式の骨格」を持つ。

### ConstraintsAST は値域・依存・意味制約を表現する

競プロドメインでは ConstraintsAST は主に以下を担う。

範囲制約
例: 1 <= N <= 2e5
要素制約
例: 0 <= A[i] <= 10^9
長さ制約
例: len(A) = N
ケース数制約
例: T 個のテストケース
将来的には関係制約
例: sum(A) <= K, 全要素相異なる, グラフが木である など
### 入力方式と制約を分離しつつ接続する

競プロ問題では

入力形式
制約

は別欄で書かれることが多いが、実体としては密接に結びついている。

例:

入力形式に A_1 ... A_N がある
制約に 0 <= A_i <= 10^9

このため設計上は

StructureAST が A という配列入力の存在を表し
ConstraintsAST が A の各要素制約を表す

という分離を維持しつつ、両者が同じ変数参照系で結びつく必要がある。

#### 研究上の主張にどう繋がるか

この競プロDSL性は、研究として次の主張に繋がる。

### ASTが人間可読仕様と具体入力生成の両方の正本になる

通常は

問題文
制約欄
サンプルケース
が別々に手管理されがちである。

本ライブラリでは、
単一のASTから

競プロサイト風の制約欄
入力方式欄
妥当なサンプルケース

を一貫して導出できることが特徴となる。

### GUIは単なる見た目ではなく、競プロ仕様構築の安全な編集器になる

GUIから構造を組み立てることは、単なる入力補助ではなく、

制約付きの仕様ASTを壊さず構築すること
その結果として表示・生成・検証可能な問題仕様を得ること

に意味がある。

### 競プロドメインは評価に適している

競プロ問題は

表示可能性
生成可能性
妥当性検査
を評価しやすい。

したがって本研究では、競プロドメインを「ニッチな応用」ではなく、
構造的仕様記述と制約付き生成を評価するための良質なベンチマーク領域として位置づける。

#### 設計資料への統合用 要約

本ライブラリは競技プログラミング問題記述DSLを初期対象とする。
その目的は、入力方式・制約・出力方式の構造化表現を StructureAST / ConstraintsAST により保持し、
そこから

競プロサイト風の「入力方式」「制約」欄を決定的に再構成し、
制約を満たすランダムサンプルケースを生成可能にする

ことである。

このとき API は競プロ利用者が認知している概念

整数
配列
長さ
値域
テストケース
に近い形で提供されるが、その実体は文字列テンプレートではなく構造化ASTである。

したがって本DSLは、競プロ問題の自然言語記述を直接扱うものではなく、
競プロ問題の形式仕様を表現・編集・表示・生成するための構造的DSLとして位置づけられる。
