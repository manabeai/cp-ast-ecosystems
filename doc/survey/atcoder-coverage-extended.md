## 3. 新規ギャップ分析

### ギャップ H: Repeat にループ変数（イテレーションインデックス）がない

- **出現問題**: ABC370-B（Binary Alchemy）
- **具体例**:

```
入力形式:
N
A_{1,1}
A_{2,1} A_{2,2}
A_{3,1} A_{3,2} A_{3,3}
...
A_{N,1} A_{N,2} ... A_{N,N}
```

行 i には i 個の要素がある下三角行列。

- **現在の限界**:
  - `Repeat.count` は `Reference`（他ノードへの参照）であり、ループインデックスを参照できない
  - `Matrix { rows, cols }` は矩形（全行同一列数）前提で、jagged 構造を表現不能
  - Gap A（`Repeat.count` が Expression 非対応）と Gap D（下三角行列非対応）の根本原因
  - たとえ Gap A を解消して Expression 対応にしても、「現在の行番号 i」を参照する仕組みがなければ表現不能

- **対応案**:

```rust
// Option 1: Repeat にループ変数を追加
Repeat {
    count: Expression,        // Gap A の解消
    index_var: Option<Name>,  // 新規: ループ変数名 (e.g., "i")
    body: Vec<NodeId>,
}

// Option 2: 専用の TriangularMatrix ノード
TriangularMatrix {
    name: Name,
    size: Reference,          // N
    triangle: Upper | Lower,
}
```

Option 1 は汎用性が高く、下三角以外の index 依存パターン（例: 行 i に N-i 個の要素がある上三角行列）にも対応可能。Option 2 は実装が簡潔だが拡張性に欠ける。

- **推奨**: Option 1。Gap A の解消と同時に `index_var` を追加すれば、Gap A・D・H を一括解消できる。

| 優先度 | ギャップ | 理由 |

| **P0** | **A** (Repeat.count → Expression) | 出現頻度最高。グラフ問題の N-1 辺パターンは ABC-D 以降で頻出。実装は Reference → Expression への型変更で中程度。 |

## 6. 問題別詳細分析

### 6.1 ❌ 対応不可の問題

#### ABC370-B — Binary Alchemy

**入力形式**:
```
N
A_{1,1}
A_{2,1} A_{2,2}
A_{3,1} A_{3,2} A_{3,3}
...
A_{N,1} A_{N,2} ... A_{N,N}
```

**分析**: 行 i に i 個の要素を持つ下三角行列。現行の `Matrix` は矩形前提、`Repeat` にはループインデックス参照機能がない。

**AST 表現の試み（不完全）**:
```
Sequence [
  Scalar(N),
  Repeat(count=N) {   // ← 各行を N 回繰り返す
    Array(A_row, length=???)  // ← 行番号 i を参照できない
  }
]
```

**ギャップ**: D（下三角行列）、H（ループ変数不在）

---

### 6.2 ⚠️ 部分対応の問題

#### ABC380-E — 1D Bucket Tool

**入力形式**:
```
N Q
query_1
...
query_Q
```
各クエリは `1 x c`（3要素）または `2 c`（2要素）。

**AST 表現（理論上可能）**:
```
Sequence [
  Tuple(N, Q),
  Repeat(count=Q) {
    Scalar(type),
    Choice(tag=type) {
      1 => Tuple(x, c),
      2 => Scalar(c)
    }
  }
]
```

**懸念点**: Choice.tag が同一 Repeat ボディ内の直前 Scalar を Reference で指す場合の動作が未検証。

---

#### ABC395-D — Pigeon Swap

**入力形式**:
```
N Q
op_1
...
op_Q
```
各 op は `1 a b`、`2 a b`、`3 a` のいずれか。

**AST 表現（理論上可能）**:
```
Sequence [
  Tuple(N, Q),
  Repeat(count=Q) {
    Scalar(op_type),
    Choice(tag=op_type) {
      1 => Tuple(a, b),
      2 => Tuple(a, b),
      3 => Scalar(a)
    }
  }
]
```

**懸念点**: ABC380-E と同様。加えて variant 1 と variant 2 のボディが同一構造だが意味が異なる点は AST 上問題ない。

### 6.3 特に注目すべき ✅ 問題

#### ABC360-C — Move It（複数独立配列）

```
N
A_1 A_2 ... A_N
W_1 W_2 ... W_N
```

同一長 N の配列が2つ並ぶ。`Sequence [Scalar(N), Array(A, N), Array(W, N)]` で表現可能。配列間の「同一インデックスで対応する」関係は AST 構造で暗黙的に表現される。

#### ABC360-D — Ghost Ants（文字列 + 配列の複合入力）

```
N T
S
X_1 X_2 ... X_N
```

S は 01 文字列で長さ N、X は整数配列で長さ N。`Tuple(N,T)`, `Scalar(S)`, `Array(X, N)` で完全表現可能。制約「S は 0 と 1 からなる長さ N の文字列」は `CharSet(S, {0, 1})` + `StringLength(S, N)` で表現。

#### ABC390-E — Vitamin Balance（Repeat 内 Tuple）

```
N X
V_1 A_1 C_1
V_2 A_2 C_2
...
V_N A_N C_N
```

各行が3要素のタプル。`Repeat(N, Tuple(V_i, A_i, C_i))` で表現可能。V_i の値域 1≤V_i≤3 は `Range` 制約で表現。

#### ABC380-D — Strange Mirroring（大きなクエリ値）

```
S
Q
K_1 K_2 ... K_Q
```

K_i ≤ 10^18。i64 の最大値 ≈ 9.2×10^18 なので範囲内。Expression.Lit の i64 制限（Gap G）には該当しない。

## 7. 制約表現に関する補足分析

入力構造とは別に、各問題の制約にも注目した結果:

| 制約パターン | 出現例 | AST 対応状況 |
|-------------|--------|-------------|
| 値範囲（1≤N≤10^5 等） | 全問題 | ✅ Range |
| 文字種制約（0/1、A/B等） | ABC360-D, ABC361-D, ARC180-A | ✅ CharSet |
| 文字列長 = 変数（\|S\|=N） | ABC360-D, ABC380-C | ✅ StringLength / LengthRelation |
| 全要素相異（X_i≠X_j） | ABC360-D | ✅ Distinct |
| 順列であること | ABC390-A (暗黙), ARC180-B | ✅ Property(permutation) |
| 整数制約 | 大半の問題 | ✅ TypeDecl(Int) |
| 文字の出現回数制約 | ABC360-A（R,M,Sが各1個） | ⚠️ Guarantee で代替 |
| H×W ≤ 定数（積の制約） | ABC370-D（H×W≤4×10^5） | ⚠️ Guarantee で代替 |
| 負の値範囲 | ABC370-E（-10^9≤A_i） | ✅ Range（負数対応済み） |

**「Guarantee で代替」パターン**: 「文字の出現回数」「変数の積の上界」などは構造化制約で直接表現できないが、`Guarantee { description }` で自由テキストとして記録可能。出現頻度が高まれば専用制約ノードの追加を検討すべき。

## 8. まとめ・推奨アクション

### 調査結果サマリ

- **31問を調査**し、28問（90%）が現行 AST で完全対応可能
- **1問（3%）が対応不可**（ABC370-B: 下三角行列）
- **2問（6%）が部分対応**（ABC380-E, ABC395-D: クエリ型分岐入力）
- **新規ギャップ 1件**を同定: Gap H（ループ変数不在）

### ABC問題の入力形式は高度に定型化されている

ABC の A〜D 問題（全体の約85%）はほぼ全て以下のパターンに集約される:

1. `Scalar(N)` + `Array(A, N)` — 最頻出
2. `Tuple(N, M, ...)` + `Array(A, N)` — 2番目に多い
3. 単一 `Scalar` / `Tuple` — A問題に多い
4. `Tuple` + `Repeat(count, Tuple)` — D問題のグラフ入力

これらは全て現行 AST で完全に表現可能であり、**ABC の A〜D 問題に限れば推定カバレッジは 95% 以上**。

### 推奨アクション

1. **最優先**: Gap A 解消（`Repeat.count` を `Expression` 化）— グラフ問題への対応に必須
2. **次優先**: Gap B の実装検証（`Choice` の `Repeat` 内ネスト動作テスト）— クエリ型問題に必須
3. **中期**: Gap H 解消（ループ変数の導入）— Gap A と同時に設計すれば効率的、Gap D も同時解消
4. **低優先**: Gap C/E/F/G — 出現頻度が低く `Guarantee` で代替可能

### 前回調査との統合カバレッジ

| | 前回(17問) | 今回(31問) | 合計(48問) |
|---|----------|----------|----------|
| ✅ 完全対応 | ~13問 | 28問 | ~41問 |
| ⚠️ 部分対応 | ~2問 | 2問 | ~4問 |
| ❌ 対応不可 | ~2問 | 1問 | ~3問 |
| **推定カバレッジ** | ~76% | **90%** | **~85%** |

Gap A と Gap B を解消すれば、推定カバレッジは **95%以上** に到達する見込み。
